## クリーンアーキテクチャの基本概念

内側の層が外部の層に依存しない設計原則に基づいている
今回の野球の観戦チケットフローで考える

- 1. Domain Layer(ドメイン層) - 最も内側の層
・ドメイン層 ビジネスのルールそのもの DBやWebには一切依存しない　ユーザーとは何か、注文とは何かというデータ構造
バリデーション
・在庫がマイナスになってはいけない。メールアドレスの形式が正しいかといった内容
・チェスで言うなら「コマの動き方」や「チェックメイト判定」がドメイン層

- 2. Application Layer(アプリケーション層)
・「事務局(調整役)」の役割　DBを意識するのではなく、「ユースケース」を実現するためにドメイン層や、インフラそうを組み合わせます

- 3. Infrastructure Layer（インフラストラクチャ層）
・SQLなどの具体的な技術詳細を書く場所
・実際のデータベースへの接続
・外部メール送信サービスや決済ゲートウェイトの通信


- 4. Presentation Layer（プレゼンテーション層）
・HTTPリクエスト(JSON)を受け取る
・JSONをGoの構造体にデコードする
・「メールアドレスの形式が正しいか」などのバリデーションを行う
・結果をHTTPステータスコードで返却する


クリーンアーキテクチャの核心は、依存関係の方向性にあります
```bash
Presentation Layer → Application Layer → Domain Layer ← Infrastructure Layer
```
・Domain層はInfrastructure層の存在を知らない
・Application層はPresentation層の詳細を知らない

依存関係は内側に向かって流れる
- 外側の層は内側の層に依存できる
- 内側の層は外側の層に依存してはいけない
